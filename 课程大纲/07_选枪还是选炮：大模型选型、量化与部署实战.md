### Part 1：选型框架——五维评估矩阵（12 min）

#### 1.1 五个核心维度（5 min）

**维度一：能力（Capability）**
- 衡量指标：MMLU（知识）、HumanEval（代码）、MATH（推理）、MT-Bench（指令跟随）
- 关键洞察：**Benchmark ≠ 实际表现**。任务相关的 benchmark 才有参考价值
- 📌 实操建议：在自己的任务上做 30-50 个样本的人工评测，比看 leaderboard 更可靠

**维度二：成本（Cost）**
- 闭源 API：按 Token 计费，单位为 $/1M tokens（Input/Output 分开计）
- 开源自部署：GPU 成本 + 运维成本（隐性成本常被低估）
- 成本公式：`月费用 = 日均请求数 × 平均tokens/请求 × 30 × 单价 × 1.2（缓冲）`

| 模型 | Input 价格 | Output 价格 | 说明 |
|------|-----------|------------|------|
| GPT-4o | $2.5/1M | $10/1M | 旗舰级，2026 最新定价 |
| GPT-4o-mini | $0.15/1M | $0.6/1M | 轻量级，性价比高 |
| Claude 3.5 Sonnet | $3/1M | $15/1M | 代码/分析能力强 |
| Llama-3-70B (自部署) | ~$0.1/1M | ~$0.1/1M | 估算值，取决于GPU利用率 |
| Qwen2.5-7B (自部署) | ~$0.01/1M | ~$0.01/1M | 极低成本，适合高频场景 |

**维度三：延迟（Latency）**
- 两个关键指标：
  - **TTFT（Time to First Token）**：用户感知延迟，目标 < 500ms
  - **TPS（Tokens Per Second）**：生成速度，流式输出时影响体验
- 影响因素：模型大小 > 量化精度 > 网络延迟 > 服务器负载

**维度四：上下文窗口（Context Window）**
- 长文档处理：需要 32K+ （如 Claude 3.5 的 200K、GPT-4 的 128K）
- 普通对话：4K-8K 完全够用
- ⚠️ 陷阱：上下文越长，成本越高，延迟越大；不要无脑选长上下文

**维度五：许可证与数据隐私（License & Privacy）**
- 数据隐私：医疗/金融/政务数据 → 必须私有化部署，闭源 API 不可用
- 商业许可：Llama-3（Meta 自定义许可）、Mistral（Apache 2.0）、Qwen（Qwen License）
- 💡 Llama-3 允许商用但禁止月活 > 7 亿的超级平台使用

#### 1.2 五维评估矩阵工具（4 min）

```python
# 选型决策工具：五维评分矩阵
model_candidates = {
    "gpt-4o": {
        "capability": 9,    # 1-10分
        "cost": 3,          # 分越高越便宜
        "latency": 7,       # 分越高越快
        "context": 9,       # 128K
        "privacy": 4,       # 数据需发送至OpenAI
    },
    "qwen2.5-7b-instruct": {
        "capability": 7,
        "cost": 9,          # 自部署极低成本
        "latency": 9,       # 小模型极快
        "context": 7,       # 32K
        "privacy": 10,      # 完全私有化
    },
    "llama-3-70b": {
        "capability": 8,
        "cost": 7,
        "latency": 6,
        "context": 7,
        "privacy": 10,
    }
}

# 场景1：高频客服问答（成本和延迟权重高）
weights_customer_service = {
    "capability": 0.20, "cost": 0.30,
    "latency": 0.25, "context": 0.10, "privacy": 0.15
}

def score_model(model_scores, weights):
    return sum(model_scores[dim] * weights[dim] for dim in weights)

for name, scores in model_candidates.items():
    print(f"{name}: {score_model(scores, weights_customer_service):.2f}")

# 输出示例（满分10）：
# gpt-4o: 6.15
# qwen2.5-7b-instruct: 8.85  ← 高频客服首选
# llama-3-70b: 7.55
```

### Part 2：量化原理——让大模型住进小显卡（10 min）

#### 2.1 为什么需要量化？（3 min）

**量化核心动机**：
- Llama-3-70B 全精度（BF16）需要 ~140GB 显存 → 需要 2-4 张 A100
- INT4 量化后仅需 ~35GB → 1 张 A100 或 2 张 RTX 3090 可跑
- Mac M2 Pro (16GB) → 可以跑 8B 模型的 Q4_K_M 量化版

**精度压缩原理**：
```
BF16（16位）: 每个权重占 2 字节 → 70B 参数 × 2B = 140GB
INT8（8位）:  每个权重占 1 字节 → 70B 参数 × 1B =  70GB
INT4（4位）:  每个权重占 0.5 字节 → 70B 参数 × 0.5B = 35GB

量化公式（INT8为例）：
  量化：q = round(x / scale) + zero_point
  反量化：x ≈ (q - zero_point) × scale

  其中 scale = (max - min) / 255，zero_point 是偏置
```

**量化误差 vs 收益权衡**：

| 量化精度 | 显存减少 | 速度提升 | 质量损失 | 推荐场景 |
|---------|---------|---------|---------|---------|
| BF16（原始） | - | 1x | 0% | 训练、高精度推理 |
| INT8 | 50% | 1.5-2x | <1% | 生产环境，质量敏感 |
| INT4 (Q4_K_M) | 75% | 2-3x | 2-5% | 资源受限，可接受轻微损失 |
| INT4 (Q2_K) | 87.5% | 3-4x | 10%+ | 仅限实验，不推荐生产 |

#### 2.2 三种量化方法对比（5 min）

**方法一：RTN（Round-to-Nearest，朴素量化）**
- 原理：直接将权重四舍五入到最近整数
- 优点：极快，无需校准数据
- 缺点：精度损失大，某些层崩溃（Outlier 问题）
- 适用：快速验证，不用于生产

**方法二：GPTQ（Generative Pre-trained Transformer Quantization）**
> 论文：Frantar et al., 2022 - "GPTQ: Accurate Post-Training Quantization for Generative Pre-trained Transformers"

- 核心思想：用 **Hessian 矩阵** 识别权重重要性，逐层、逐列量化
- 关键技术：
  1. Optimal Brain Quantization（OBQ）框架：最小化每层的量化误差
  2. 贪婪列更新：每量化一列权重，用 Hessian 补偿其他列的误差
  3. 利用 GPU 并行化：将 OBQ 扩展到 1000 亿参数级别
- 特点：需要少量校准数据（约 128 条样本），离线处理，结果固定
- 精度：接近 BF16，INT4 量化 GPT-3.5 级模型误差 < 2%

**方法三：AWQ（Activation-aware Weight Quantization）**
> 论文：Lin et al., 2023 - "AWQ: Activation-aware Weight Quantization for LLM Compression and Acceleration"

- 核心洞察：**权重不是同等重要的**——激活值较大的通道（约 1%）对精度贡献超过 99%
- 关键技术：
  1. 分析输入激活统计量，识别"关键通道"
  2. 对关键通道的权重做 per-channel scaling（保护重要权重）
  3. 其余权重正常 INT4 量化
- 优势：比 GPTQ **推理速度更快**（因为内存访问模式更友好），且精度相当
- 2026 现状：AWQ 是主流推理框架（vLLM/TGI）的首选量化格式

**方法四：GGUF（llama.cpp 格式）**
- 不是量化方法，而是**量化模型的存储格式**（原 GGML → GGUF）
- 命名规则：`Llama-3-8B-Instruct-Q4_K_M.gguf`
  - `Q4`：INT4 量化
  - `K`：使用 K-quants（混合精度，关键层用更高精度）
  - `M`：Medium（权衡精度和速度，推荐选项）
- 优势：支持 CPU+GPU 混合推理，可在没有 CUDA 的机器上运行

#### 2.3 量化选择决策树（2 min）

```
有 GPU（CUDA）？
    │ 是                      │ 否（Mac/CPU）
    ▼                         ▼
需要最高推理速度？          使用 GGUF（llama.cpp/Ollama）
    │ 是           │ 否       选 Q4_K_M（推荐） 或 Q5_K_M（更高精度）
    ▼             ▼
  AWQ          GPTQ
(vLLM优化)   (HuggingFace
             AutoGPTQ)
```

**⚠️ 面试考点**：GPTQ 和 AWQ 的区别——GPTQ 基于 Hessian 逐列补偿误差，AWQ 基于激活感知保护关键通道，AWQ 推理更快但需要激活统计分析。

### Part 3：部署实战——三条路线代码示例（15 min）

#### 3.1 路线 A：云 API（最简路线，5 min）

**适用场景**：快速原型 / 团队无 GPU / 数据合规允许出境

```python
# 路线A：统一接口调用多个云API（使用 OpenAI 兼容接口）
from openai import OpenAI

# OpenAI
client_openai = OpenAI(api_key="YOUR_KEY")

# Anthropic（Claude，通过 OpenAI 兼容接口）
client_anthropic = OpenAI(
    api_key="YOUR_ANTHROPIC_KEY",
    base_url="https://api.anthropic.com/v1/"
)

# 阿里云灵积（Qwen，国内合规）
client_qwen = OpenAI(
    api_key="YOUR_DASHSCOPE_KEY",
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1"
)

def call_model(client, model: str, prompt: str) -> dict:
    """统一调用接口，返回内容和Token用量"""
    response = client.chat.completions.create(
        model=model,
        messages=[{"role": "user", "content": prompt}],
        temperature=0.7,
        max_tokens=512
    )
    return {
        "content": response.choices[0].message.content,
        "input_tokens": response.usage.prompt_tokens,
        "output_tokens": response.usage.completion_tokens,
        # 成本估算（以GPT-4o为例）
        "cost_usd": (response.usage.prompt_tokens * 2.5 +
                     response.usage.completion_tokens * 10) / 1_000_000
    }

# 对比三个模型
prompt = "用三句话解释 Transformer 的核心创新"
for name, (client, model) in {
    "GPT-4o": (client_openai, "gpt-4o"),
    "GPT-4o-mini": (client_openai, "gpt-4o-mini"),
    "Qwen2.5-72B": (client_qwen, "qwen2.5-72b-instruct"),
}.items():
    result = call_model(client, model, prompt)
    print(f"\n[{name}] 成本: ${result['cost_usd']:.5f}")
    print(f"回复: {result['content'][:100]}...")
```

#### 3.2 路线 B：vLLM 自部署（生产级，5 min）

**适用场景**：高并发生产环境 / 有 GPU 服务器 / 需要控制成本

**核心技术：PagedAttention**
> vLLM 论文：Kwon et al., 2023 - "Efficient Memory Management for Large Language Model Serving with PagedAttention"

- 传统 KV Cache：为每个请求预分配连续最大内存（巨大浪费）
- PagedAttention：借鉴操作系统虚拟内存分页，按需分配 KV Cache 块
- 效果：GPU 内存利用率从 ~40% 提升到 ~90%，吞吐量提升 2-4 倍

```bash
# 安装并启动 vLLM 服务（兼容 OpenAI API 格式）
pip install vllm

# 启动服务（Qwen2.5-7B AWQ量化版）
python -m vllm.entrypoints.openai.api_server \
    --model Qwen/Qwen2.5-7B-Instruct-AWQ \
    --quantization awq \
    --max-model-len 8192 \
    --gpu-memory-utilization 0.9 \
    --tensor-parallel-size 1 \
    --port 8000
```

```python
# 调用 vLLM 服务（与 OpenAI 完全兼容）
from openai import OpenAI
import asyncio, time

client = OpenAI(base_url="http://localhost:8000/v1", api_key="dummy")

# 测试单次调用
response = client.chat.completions.create(
    model="Qwen/Qwen2.5-7B-Instruct-AWQ",
    messages=[{"role": "user", "content": "你好！介绍一下自己"}],
    stream=True  # 流式输出
)

# 流式接收
start = time.time()
full_text = ""
for chunk in response:
    if chunk.choices[0].delta.content:
        full_text += chunk.choices[0].delta.content
        print(chunk.choices[0].delta.content, end="", flush=True)

elapsed = time.time() - start
print(f"\n\n速度: {len(full_text.split()) / elapsed:.1f} tokens/s")

# 📊 典型性能（RTX 3090 24GB，Qwen2.5-7B-AWQ）：
# 单请求：~80-100 tokens/s
# 并发16请求：~400 tokens/s 总吞吐（连续批处理优化）
```

#### 3.3 路线 C：Ollama 本地部署（最简单，5 min）

**适用场景**：个人开发 / 原型验证 / 离线场景 / Mac M系列芯片

```bash

brew install ollama
```

### Part 4：产品视角——成本优化三板斧（13 min）

#### 4.1 成本估算公式与基准测算（4 min）

**Token 成本模型**：

```
月总成本 = Σ(场景_i 的日均请求数 × 平均Token/请求 × 30天 × 单价) × 1.2

其中：
  平均Token/请求 = system_prompt_tokens + avg_user_tokens + avg_output_tokens
  1.2 = 安全系数（考虑峰值和计费误差）
```

**实战案例：企业客服 AI 成本测算**

```python
# 成本估算器
class CostEstimator:
    # 2026年参考价格（$/1M tokens）
    PRICING = {
        "gpt-4o":         {"input": 2.5,  "output": 10.0},
        "gpt-4o-mini":    {"input": 0.15, "output": 0.6},
        "claude-3.5-sonnet": {"input": 3.0, "output": 15.0},
        "qwen2.5-72b-api": {"input": 0.4, "output": 1.2},  # 阿里云
        "self-hosted-7b": {"input": 0.01, "output": 0.01}, # 估算
    }

    def estimate_monthly(self, model: str, daily_requests: int,
                         system_tokens: int, avg_user_tokens: int,
                         avg_output_tokens: int) -> dict:
        pricing = self.PRICING[model]
        monthly_input  = daily_requests * (system_tokens + avg_user_tokens) * 30
        monthly_output = daily_requests * avg_output_tokens * 30

        cost = (monthly_input  * pricing["input"] / 1_000_000 +
                monthly_output * pricing["output"] / 1_000_000) * 1.2

        return {
            "model": model,
            "monthly_input_tokens":  monthly_input,
            "monthly_output_tokens": monthly_output,
            "monthly_cost_usd": round(cost, 2),
            "monthly_cost_cny": round(cost * 7.2, 2)
        }

estimator = CostEstimator()

# 场景：客服AI，日均1万次对话
# system_prompt=500, 用户平均150 tokens, 回复平均200 tokens
scenarios = {
    "GPT-4o":         ("gpt-4o",         10000, 500, 150, 200),
    "GPT-4o-mini":    ("gpt-4o-mini",    10000, 500, 150, 200),
    "Qwen2.5-72B(云)":("qwen2.5-72b-api",10000, 500, 150, 200),
    "自部署7B":       ("self-hosted-7b", 10000, 500, 150, 200),
}

print(f"{'方案':<20} {'月成本(USD)':<15} {'月成本(CNY)':<15}")
print("-" * 50)
for name, args in scenarios.items():
    result = estimator.estimate_monthly(*args)
    print(f"{name:<20} ${result['monthly_cost_usd']:<14} ¥{result['monthly_cost_cny']}")

# 输出示例（10万次/日，500+150+200=850 tokens/次）：
# GPT-4o:          $3,240      ¥23,328
# GPT-4o-mini:     $194        ¥1,397
# Qwen2.5-72B(云): $259        ¥1,865
# 自部署7B:        $9          ¥65（月均GPU摊销约800元额外）
```

#### 4.2 降本策略一：Prompt Cache（3 min）

**原理**：如果多个请求共享相同前缀（如 System Prompt），可以缓存前缀的 KV 计算结果，后续请求直接复用。

**效果**：有长 System Prompt（500+ tokens）时，可节省 40-60% 的输入 Token 成本。

**支持情况**（2026年）：
- Anthropic Claude：自动开启 Prompt Caching（前缀 > 1024 tokens 自动缓存）
- OpenAI GPT-4o：自动 Prompt Caching（相同前缀 > 1024 tokens）
- vLLM：通过 `prefix_caching=True` 开启

```python
# 验证Prompt Cache效果（Anthropic API）
import anthropic, time

client = anthropic.Anthropic()
system_prompt = "你是一个专业的法律顾问..." + "（假设这是500 tokens的长系统提示）" * 20

# 第一次调用（无缓存）
start = time.time()
response1 = client.messages.create(
    model="claude-3-5-sonnet-20241022",
    max_tokens=100,
    system=[{"type": "text", "text": system_prompt,
              "cache_control": {"type": "ephemeral"}}],  # 标记为可缓存
    messages=[{"role": "user", "content": "合同违约有哪些法律后果？"}]
)
t1 = time.time() - start
print(f"第一次（冷启动）: {t1:.2f}s, 缓存写入: {response1.usage.cache_creation_input_tokens} tokens")

# 第二次调用（命中缓存）
start = time.time()
response2 = client.messages.create(
    model="claude-3-5-sonnet-20241022",
    max_tokens=100,
    system=[{"type": "text", "text": system_prompt,
              "cache_control": {"type": "ephemeral"}}],
    messages=[{"role": "user", "content": "劳动合同解除的条件是什么？"}]
)
t2 = time.time() - start
print(f"第二次（命中缓存）: {t2:.2f}s, 缓存读取: {response2.usage.cache_read_input_tokens} tokens")
print(f"成本节省: ~{response2.usage.cache_read_input_tokens * 0.1 / 1000:.4f}$ (缓存读取按10%计费)")
# 缓存命中后：延迟减少30-50%，成本减少约90%（cached tokens按原价10%计费）
```

#### 4.3 降本策略二：模型路由（3 min）

**核心思想**：不同复杂度的任务路由到不同成本的模型，"杀鸡不用牛刀"。

```python
# 简单的基于规则的模型路由器
class ModelRouter:
    """根据任务复杂度路由到合适模型"""

    SIMPLE_KEYWORDS = ["是什么", "定义", "翻译", "摘要", "格式化"]
    COMPLEX_KEYWORDS = ["分析", "推理", "代码", "方案", "对比", "评估"]

    def route(self, user_message: str, context_length: int) -> str:
        """返回推荐模型名称"""
        # 超长上下文必须用支持长窗口的模型
        if context_length > 32000:
            return "claude-3-5-sonnet"  # 200K context

        msg_lower = user_message.lower()
        complexity_score = 0

        # 简单任务信号
        for kw in self.SIMPLE_KEYWORDS:
            if kw in msg_lower:
                complexity_score -= 1

        # 复杂任务信号
        for kw in self.COMPLEX_KEYWORDS:
            if kw in msg_lower:
                complexity_score += 2

        # 长消息倾向于复杂任务
        if len(user_message) > 500:
            complexity_score += 1

        # 路由决策
        if complexity_score >= 3:
            return "gpt-4o"           # 复杂任务：旗舰模型
        elif complexity_score >= 1:
            return "gpt-4o-mini"      # 中等任务：轻量模型
        else:
            return "qwen2.5-7b-local" # 简单任务：本地模型

    def estimate_savings(self, routing_distribution: dict) -> float:
        """估算相比全用GPT-4o的成本节省率"""
        # 假设平均每次请求成本（相对值）
        model_costs = {"gpt-4o": 1.0, "gpt-4o-mini": 0.06, "qwen2.5-7b-local": 0.004}
        weighted_cost = sum(
            pct * model_costs.get(model, 1.0)
            for model, pct in routing_distribution.items()
        )
        return (1 - weighted_cost) * 100

router = ModelRouter()

# 测试路由
test_cases = [
    ("翻译这段英文：Hello World", 500),
    ("分析以下代码的时间复杂度并给出优化方案...", 1200),
    ("总结这篇文章的主要观点", 200),
]

for msg, ctx_len in test_cases:
    model = router.route(msg, ctx_len)
    print(f"任务: {msg[:30]}... → 路由到: {model}")

# 典型路由分布（客服场景）：
# 60% 简单问答 → 本地7B
# 30% 中等任务 → GPT-4o-mini
# 10% 复杂分析 → GPT-4o
distribution = {"qwen2.5-7b-local": 0.6, "gpt-4o-mini": 0.3, "gpt-4o": 0.1}
savings = router.estimate_savings(distribution)
print(f"\n相比全用GPT-4o，智能路由可节省约 {savings:.1f}% 成本")
# 输出：相比全用GPT-4o，智能路由可节省约 95.6% 成本
```

#### 4.4 降本策略三：批处理与异步化（3 min）

**原理**：
- **Batch API（OpenAI/Anthropic）**：非实时任务打包发送，价格折半（50% off）
- **离线批处理**：数据分析、报告生成等无需实时响应的任务批量处理

```python
# OpenAI Batch API：适合离线任务（价格折半，24小时内返回）
import json
from openai import OpenAI

client = OpenAI()

# 准备批量请求
batch_requests = []
for i, text in enumerate(documents):  # documents是待处理文档列表
    batch_requests.append({
        "custom_id": f"doc-{i}",
        "method": "POST",
        "url": "/v1/chat/completions",
        "body": {
            "model": "gpt-4o-mini",
            "messages": [
                {"role": "system", "content": "提取文档中的关键信息，以JSON格式返回"},
                {"role": "user", "content": text}
            ],
            "max_tokens": 500
        }
    })

# 写入JSONL文件并上传
with open("batch_requests.jsonl", "w") as f:
    for req in batch_requests:
        f.write(json.dumps(req, ensure_ascii=False) + "\n")

batch_input = client.files.create(
    file=open("batch_requests.jsonl", "rb"),
    purpose="batch"
)

# 创建批量任务（24小时内完成，价格是实时API的50%）
batch = client.batches.create(
    input_file_id=batch_input.id,
    endpoint="/v1/chat/completions",
    completion_window="24h"
)

print(f"批量任务已创建: {batch.id}")
print(f"预估节省: {len(batch_requests) * 0.5 * 200 * 0.6 / 1_000_000:.4f}$ (50% 折扣)")
# 适合场景：离线文档处理、批量数据标注、定时报告生成
```

#### 5.3 面试真题与参考答案

**🔥 高频真题 1**：如何选择开源模型还是闭源 API？
> 参考框架：数据隐私（是否允许数据出境）→ 成本规模（日均调用量）→ 运维能力（有无 GPU 和工程师）→ 质量要求（benchmark 验证）。数据敏感或高频调用推荐开源自部署；快速原型或低频高质量需求推荐闭源 API。

**🔥 高频真题 2**：解释 GPTQ 和 AWQ 的区别
> GPTQ 用 Hessian 矩阵逐列量化，在列量化后用矩阵补偿残差，精度高但推理相对较慢；AWQ 识别关键激活通道并保护其高精度，其余 INT4 量化，推理更快，是 vLLM 等生产框架的首选。

**⚠️ 中频真题 3**：vLLM 为什么比 HuggingFace Transformers 推理快？
> PagedAttention：将 KV Cache 分页管理（类似操作系统虚拟内存），消除内存碎片，GPU 利用率从 40% 提升到 90%；连续批处理：动态打包不同长度请求，而非等待固定批次，吞吐量提升 2-4 倍。

**💡 进阶真题 4**：如果你有 10 万元预算部署一个企业 AI 客服，你会怎么做？
> 参考回答方向：预算分配（GPU 服务器 vs 云 API）、模型选择（Qwen2.5-72B 或 Llama-3-70B）、量化方案（AWQ INT4）、部署工具（vLLM）、降本策略（Prompt Cache + 路由）、监控与评估。
