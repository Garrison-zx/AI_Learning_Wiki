## 二、课程简介

"大力出奇迹"是 AI 圈最流行的一句话，也是最容易被误解的一句话。

它的正确含义是：在语言模型这个特定任务上，增大规模（参数量 × 数据量 × 算力）能够可预测地提升性能，并在某些阈值处涌现出没人设计过的新能力。

它的错误解读是：只要堆参数就行。现实是——1750 亿参数的 GPT-3 被 700 亿参数的 Chinchilla 在多个任务上击败，因为 GPT-3 数据严重不足；而 1750 亿参数的模型如果没有 MoE、GQA、FlashAttention 等工程优化，根本无法被实际训练和部署。

本课从"大力"的科学依据出发，深入解析使大模型成为可能的架构创新，并坦诚地面对其边界。

### 🎯 Part 1：直觉导入（8min）

#### 1.1 一句话理解"大力出奇迹"

> **核心命题**：在语言模型上，规模（参数 × 数据 × 算力）与性能之间存在**可预测的幂律关系**，且在特定阈值处会涌现出没有被显式训练的新能力。

简化版：**更大 → 更好 → 某个节点突然会做以前完全不会的事。**

**但有两个重要限定**：
- "大"不只是参数量，数据量同样关键（Chinchilla 修正）
- "大力"需要工程支撑——没有 MoE/GQA/FlashAttn，1750 亿参数根本跑不起来

#### 1.2 生活类比：城市规模效应

> 城市经济学发现：城市人口每翻一倍，人均 GDP 增长约 15%，专利产出增长约 20%——规模带来的不只是线性增长，而是超线性回报（Bettencourt et al., 2007）。

语言模型和城市有惊人的相似性：
- 城市里有更多人 → 更多协作可能 → 非线性创新涌现
- 模型里有更多参数 → 更复杂的特征交互 → 非线性能力涌现
- **城市会涌现出新的职业**（程序员、UX 设计师在 100 年前不存在）
- **模型会涌现出新的能力**（数学推理、代码生成在设计时未被预期）

城市也有极限：交通拥堵、房价、污染——大模型同样有数据墙、计算墙、对齐墙。

> 💡 **互动题**：GPT-3 和 GPT-2 参数量相差约 100 倍，但 GPT-3 的 ICL 能力是从 GPT-2 的 0 跳跃到了稳定可用——这是"10 倍更好"还是"完全不同的能力"？（先想，再看 2.3 节）

### 🔬 Part 2：颗粒级原理拆解（28min）

#### 2.1 什么叫"大"：规模的三个维度（3min）

"大模型"的"大"不只是参数量，是三个维度同时放大：

```
规模三角形

        参数量 N
       （模型容量）
           /\
          /  \
         /    \
        /  算力  \
       /    C    \
      /____________\
   数据量 D        算力 C
  （训练 token 数） （FLOPs）
```

| 维度 | GPT-2（2019） | GPT-3（2020） | LLaMA 3（2024） |
|------|-------------|-------------|----------------|
| 参数量 N | 15亿 | 1750亿 | 4050亿（最大版） |
| 训练数据 D | 40GB（WebText） | 570GB（混合） | 15万亿 tokens |
| 算力 C（估算） | ~1×10²¹ FLOPs | ~3×10²³ FLOPs | ~3×10²⁴ FLOPs |

**关键规律**：三个维度的提升是**协同的**——只扩大参数量而不增加数据，效果不如均衡扩大。这是 Chinchilla 的核心发现。

#### 2.2 Scaling Law：让"大力"可预测（六层追问）（8min）

| 层 | 问题 | 回答 |
|----|------|------|
| **第 1 层** | Scaling Law 是什么？ | 语言模型的验证集损失 L 与参数量 N、数据量 D、计算量 C 之间存在稳定的幂律关系（power law）：增大任意一个维度，损失按可预测的比例下降 |
| **第 2 层** | 为什么需要 Scaling Law？ | 训练 GPT-3 这样的模型成本数百万美元，不能靠试错决定参数量。Scaling Law 让研究者能**在小规模实验上预测大规模模型的性能**，从而合理分配算力预算 |
| **第 3 层** | Kaplan 2020 的结论是什么？ | 固定算力预算 C 时，大参数 + 少数据（少训练步数）优于小参数 + 多数据。建议把大部分预算给参数量 `N ∝ C^{0.73}`，数据量 `D ∝ C^{0.27}` |
| **第 4 层** | Chinchilla（2022）为什么推翻了 Kaplan？ | Chinchilla 发现 Kaplan 的实验训练时间太短——实验中"增加参数比增加数据有效"是因为数据没有训练够。修正后的最优比例：**参数量和数据量大致相等**（按 token 估算：训练 token 数 ≈ 参数量 × 20）。700亿参数的 Chinchilla 用 1.4万亿 tokens 训练，超过了训练了 3000 亿 tokens 的 GPT-3（1750亿）|
| **第 5 层** | Scaling Law 的数学形式？ | `L(N, D) = E + A/N^α + B/D^β` 其中 E 是不可减少的损失（数据中的噪声），A/B 是系数，α≈0.34，β≈0.28。核心：损失下降是 N 和 D 的幂函数之和 |
| **第 6 层** | Scaling Law 的局限与演进？ | ① 只在特定任务范围内预测准确，涌现能力无法预测 ② Chinchilla 适用于"计算最优"训练，但推理时更小模型更便宜，导致实践中倾向训练"过多数据"的小模型（如 LLaMA 用更多数据训练较小模型）③ 2024 年：合成数据、数据质量过滤使 Scaling Law 关系发生偏移 |

**Chinchilla 最优训练配比**（实用查询表）：

| 目标参数量 | 最优训练数据量 | 代表模型 |
|---------|-------------|---------|
| 70 亿（7B） | 1400 亿 tokens | LLaMA 2-7B |
| 130 亿（13B） | 2600 亿 tokens | LLaMA 2-13B |
| 700 亿（70B） | 1.4 万亿 tokens | Chinchilla-70B |
| 1750 亿（175B） | 3.5 万亿 tokens | GPT-3（实际只训练了 3000 亿，不足） |

> **🔥 面试高频**：LLaMA 为什么用 70B 参数训练 2 万亿 tokens，而不按 Chinchilla 配比的 1.4T tokens 停下来？
> 答：**推理成本**。训练时多消耗算力，换来更小的模型（推理速度更快、部署成本更低）——在模型要被大量部署时，过训练小模型反而更经济。

#### 2.3 涌现能力：让"大力"出奇迹（4min）

**涌现的定义与特征**：

| 特征 | 说明 |
|------|------|
| 不可预测 | 损失曲线平滑下降，但特定任务准确率在阈值前接近随机，阈值后突然跳跃 |
| 不可外推 | 小模型的性能趋势无法预测大模型的涌现点 |
| 任务多样 | 同一个语言模型，在数学、代码、多语言、推理等完全不同的任务上都会涌现 |

**已记录的涌现能力（Wei et al., 2022）**：

| 能力 | 涌现规模阈值（估计） | 具体表现 |
|------|------------------|---------|
| 算术（3位数加法） | ~13B 参数 | 小模型随机猜，13B 以上稳定正确 |
| In-Context Learning | ~100B 参数 | Few-shot 能力涌现 |
| Chain-of-Thought | ~100B+ 参数 | 逐步推理能力涌现，小模型反而会因 CoT 变差 |
| 指令跟随 | ~60-100B 参数 | 理解并执行复杂自然语言指令 |
| 多语言翻译（无监督） | ~175B 参数 | 没有专门翻译训练，仍能翻译 |

**涌现的两种主流理论**（学界尚未完全达成共识）：

```
理论 A：阈值理论
  模型内部需要"足够多的神经元协作"才能完成某类推理
  参数不足时，协作链条断裂，任务失败
  超过阈值时，完整协作链出现，任务突然成功

理论 B：测量假象论（Schaeffer et al., 2023）
  部分"涌现"是评估指标的非线性导致的
  用连续指标（如 token 概率）衡量，涌现消失，变成平滑提升
  ——这个反驳引发了学界对涌现定义的重新审视
```

> **⚠️ 易错提醒**：涌现不等于"模型突然变聪明了"。它指的是**特定任务上的能力跳跃**，模型的整体损失始终是平滑下降的。同时，部分涌现可能是评估方法的产物，而非真实能力跳跃。

##### 2.4.1 MoE（混合专家模型）：花同样的钱买 8 倍参数

| 小节 | 具体内容 | 引用 |
|-----|---------|------|
| 核心思路 | Dense 模型每次推理激活所有参数。MoE 把 FFN 层替换为 N 个"专家"网络 + 一个门控网络，每次只激活 Top-K 个专家（通常 K=2） | [Shazeer et al., 2017] |
| 计算优势 | 总参数是 Dense 的 N 倍，但每次推理的 FLOPs 仅是 1 个 Dense 模型的 K/N 倍。DeepSeek-V3 总参数 671B，每次激活 37B——等效 Dense 模型的推理成本 | [DeepSeek-V3 报告, 2024] |
| 门控机制 | `gate(x) = softmax(W_g · x)`，选取 Top-2 得分最高的专家，对其输出加权求和 | |
| 工程挑战 | 负载均衡问题：所有 token 可能都选同 2 个专家，其他专家白占显存。解决方案：辅助损失（auxiliary loss）惩罚不均衡 | [Switch Transformer, 2021] |
| 代表模型 | GPT-4（推测 MoE）、Mixtral 8×7B、DeepSeek-V3（671B/37B）、Qwen2-57B-A14B | |

```
MoE 结构示意（对比 Dense FFN）

Dense FFN：            MoE FFN：
输入 x                 输入 x
  ↓                      ↓
FFN（全量激活）          门控网络（选 Top-2 专家）
  ↓                      ↓        ↓
输出                   专家1    专家3   （共 8 个专家，只激活 2 个）
                         ↓        ↓
                      加权求和 → 输出
```

##### 2.4.2 GQA（分组查询注意力）：让推理不再被显存卡脖子

| 小节 | 具体内容 | 引用 |
|-----|---------|------|
| 问题根源 | 标准 Multi-Head Attention：h 个 Query 头 = h 个 K 头 = h 个 V 头。推理时每个头的 K/V 都要缓存在 KV Cache 中，显存占用随序列长度线性增长 | |
| MQA（多查询注意力） | 所有 Query 头共享**同一个** K 头和 V 头。显存大幅减少，但质量下降明显 | [Shazeer, 2019] |
| GQA（分组查询注意力） | **折中方案**：h 个 Query 头分成 G 组，每组共享 1 个 K/V 头（G 个 K/V 头，远少于 h 个）。G=h 时退化为 MHA，G=1 时退化为 MQA | [Ainslie et al., 2023] |
| 实际效果 | LLaMA 3（8B）：32 个 Query 头，8 个 KV 头（GQA，G=8）。KV Cache 显存减少 4 倍，推理质量接近 MHA | [LLaMA 3 技术报告, 2024] |

```
三种 Attention 的 KV 头数对比（以 32 个 Query 头为例）

MHA（标准）: Q×32 K×32 V×32  → KV Cache 最大，质量最好
GQA（分组）: Q×32 K×8  V×8   → KV Cache 减少 4 倍，质量接近 MHA  ✅ 现代首选
MQA（多查询）: Q×32 K×1  V×1  → KV Cache 最小，质量下降明显
```

##### 2.4.3 RoPE：让模型"记住"更长的上下文

| 小节 | 具体内容 | 引用 |
|-----|---------|------|
| 旋转位置编码原理 | 把位置信息编码为 Q/K 向量在复数空间的旋转角度。位置 m 的向量乘以旋转矩阵 `R_m`，使得 `Q_m · K_n = f(m-n)`——点积天然反映相对位置差 | [Su et al., 2021] |
| 为什么有利于长上下文 | 相对位置信息直接内嵌在 Attention 计算中，而不是加在 Embedding 上。外推时，模型看到"位置差为 5000"和"位置差为 500"的区别，而不是看到"绝对位置 5000"这个训练时没见过的数字 | |
| 上下文扩展技术 | YaRN（2023）：通过调整 RoPE 的频率基数（base frequency），将上下文从 4K 扩展到 128K+，无需从零重新训练 | [Peng et al., 2023] YaRN 论文 |
| 各模型上下文长度 | GPT-4 Turbo: 128K / Claude 3.5: 200K / Gemini 1.5 Pro: 1M / LLaMA 3.1: 128K | 均基于 RoPE 改进 |

---

##### 2.4.4 FlashAttention：IO 优化让训练速度翻倍

| 小节 | 具体内容 | 引用 |
|-----|---------|------|
| 瓶颈识别 | 标准 Attention 需要存储完整的 N×N 注意力矩阵到 HBM（高带宽显存）中，HBM 带宽是主要瓶颈（不是计算速度） | [Dao et al., 2022] |
| 核心优化：分块计算（Tiling） | 把 Q/K/V 分成小块，在 SRAM（片上缓存，带宽 19× HBM）中完成 Attention 计算，只把最终结果写回 HBM。永远不存完整的 N×N 矩阵 | [Dao et al., 2022] |
| 效果 | FlashAttention-2（2023）：训练速度提升 2-4 倍，内存占用从 O(N²) 降至 O(N)。已成为所有主流训练框架的标配 | [Dao, 2023] |
| FlashAttention-3 | 针对 H100 GPU 的进一步优化，利用异步执行，在 FP8 精度下速度可达理论峰值的 75% | [Shah et al., 2024] |

---

##### 2.4.5 KV Cache：推理时不做重复计算

| 小节 | 具体内容 |
|-----|---------|
| 问题 | 自回归生成：生成第 t 个 token 时，需要对前 t-1 个 token 也做 Self-Attention。如果不缓存，每步都要重算所有历史 token 的 K/V |
| KV Cache 原理 | 把每一层每个 token 的 K/V 矩阵缓存下来，生成新 token 时直接读取，只计算新 token 的 K/V |
| 显存代价 | 每个 token 的 KV Cache 大小 = `2 × num_layers × num_kv_heads × head_dim × dtype_bytes`。LLaMA 3-8B 在 128K 上下文时，KV Cache 约占 32GB |
| 优化方向 | Paged Attention（vLLM）：像操作系统内存分页一样管理 KV Cache，支持更多并发请求；量化 KV Cache（INT8/INT4），用精度换显存 |

**五大创新对照总结**：

| 技术 | 解决什么问题 | 效果量级 |
|------|-----------|---------|
| MoE | 参数量 vs 计算量的矛盾 | 同算力下参数量 ×8 |
| GQA | 推理时 KV Cache 显存过大 | KV Cache ÷4，质量基本不变 |
| RoPE | 长上下文的位置外推 | 支持 128K-1M token |
| FlashAttention | 训练时 Attention 的 IO 瓶颈 | 训练速度 ×2-4 |
| KV Cache | 自回归推理的重复计算 | 推理速度提升 10-100× |

### 💻 Part 3：代码实践（12min）

#### 3.1 手动实现 KV Cache，对比推理速度

```python
"""
功能：对比有无 KV Cache 的自回归推理速度差异
依赖：torch >= 2.0
输入：序列长度、模型层数
输出：两种方式的耗时对比
预计运行：< 5 秒（CPU）
"""
import torch
import time

def attention_without_cache(queries, keys, values):
    """无 KV Cache：每次从头计算所有 token 的 Attention"""
    scale = queries.shape[-1] ** -0.5
    scores = torch.matmul(queries, keys.transpose(-2, -1)) * scale
    weights = torch.softmax(scores, dim=-1)
    return torch.matmul(weights, values)

def simulate_generation(seq_len: int, d_model: int = 64, use_cache: bool = True):
    """
    模拟自回归生成 seq_len 个 token
    有 Cache：每步只计算新 token，复杂度 O(n)
    无 Cache：每步重算所有历史，复杂度 O(n²)
    """
    kv_cache_k = []  # 缓存历史 Key
    kv_cache_v = []  # 缓存历史 Value
    total_flops = 0

    start = time.time()
    for step in range(seq_len):
        # 当前步的新 token
        new_q = torch.randn(1, 1, d_model)       # (batch=1, seq=1, d_model)
        new_k = torch.randn(1, 1, d_model)
        new_v = torch.randn(1, 1, d_model)

        if use_cache:
            # 有 Cache：只需对新 token 做 Attention（查历史 KV）
            kv_cache_k.append(new_k)
            kv_cache_v.append(new_v)
            all_k = torch.cat(kv_cache_k, dim=1)  # (1, step+1, d_model)
            all_v = torch.cat(kv_cache_v, dim=1)
            # 只计算 1 个 Query 与 step+1 个 Key 的 Attention
            _ = attention_without_cache(new_q, all_k, all_v)
            total_flops += (step + 1)  # O(n) 累加
        else:
            # 无 Cache：每步重建全量序列再计算
            all_q = torch.randn(1, step + 1, d_model)
            all_k = torch.randn(1, step + 1, d_model)
            all_v = torch.randn(1, step + 1, d_model)
            _ = attention_without_cache(all_q, all_k, all_v)
            total_flops += (step + 1) ** 2  # O(n²) 每步

    elapsed = time.time() - start
    return elapsed, total_flops

# 对比测试
seq_len = 200
t_cached, f_cached = simulate_generation(seq_len, use_cache=True)
t_nocache, f_nocache = simulate_generation(seq_len, use_cache=False)

print(f"序列长度: {seq_len} tokens")
print(f"有 KV Cache: {t_cached:.3f}s | 计算量: {f_cached:,}")
print(f"无 KV Cache: {t_nocache:.3f}s | 计算量: {f_nocache:,}")
print(f"速度加速比: {t_nocache / t_cached:.1f}x")
print(f"计算量减少: {f_nocache / f_cached:.1f}x")
# 预期输出：无 Cache 比有 Cache 慢约 seq_len/3 倍
```

#### 3.2 MoE 门控机制的简单实现

```python
"""
功能：实现 MoE 的门控机制，演示专家选择过程
依赖：torch >= 2.0
输入：(batch_size=4, seq_len=5, d_model=64)
输出：每个 token 选择哪 2 个专家，加权后的输出
预计运行：< 1 秒（CPU）
"""
import torch
import torch.nn as nn
import torch.nn.functional as F

class SimpleMoE(nn.Module):
    def __init__(self, d_model=64, num_experts=8, top_k=2):
        super().__init__()
        self.num_experts = num_experts
        self.top_k = top_k

        # 门控网络：输出每个专家的得分
        self.gate = nn.Linear(d_model, num_experts, bias=False)

        # 8 个专家，每个都是一个简单的 FFN
        self.experts = nn.ModuleList([
            nn.Sequential(
                nn.Linear(d_model, d_model * 4),
                nn.GELU(),
                nn.Linear(d_model * 4, d_model)
            )
            for _ in range(num_experts)
        ])

    def forward(self, x):
        # x: (batch, seq_len, d_model)
        B, T, D = x.shape
        x_flat = x.view(-1, D)          # (B*T, d_model)

        # 门控得分
        gate_logits = self.gate(x_flat)  # (B*T, num_experts)
        gate_scores = F.softmax(gate_logits, dim=-1)

        # 选 Top-K 个专家
        top_k_scores, top_k_indices = torch.topk(gate_scores, self.top_k, dim=-1)
        # top_k_scores: (B*T, top_k)，归一化使 k 个权重和为 1
        top_k_scores = top_k_scores / top_k_scores.sum(dim=-1, keepdim=True)

        # 只对选中的专家做计算（稀疏激活）
        output = torch.zeros_like(x_flat)
        for k in range(self.top_k):
            expert_idx = top_k_indices[:, k]   # (B*T,) 每个 token 选的第 k 个专家
            weight = top_k_scores[:, k]        # (B*T,) 对应权重

            # 按专家分组计算（真实实现会并行）
            for e in range(self.num_experts):
                mask = (expert_idx == e)       # 选了专家 e 的 token
                if mask.any():
                    expert_out = self.experts[e](x_flat[mask])
                    output[mask] += weight[mask].unsqueeze(-1) * expert_out

        return output.view(B, T, D), top_k_indices.view(B, T, self.top_k)

# 测试
moe = SimpleMoE(d_model=64, num_experts=8, top_k=2)
x = torch.randn(4, 5, 64)  # 4个样本，每个5个token
out, expert_choices = moe(x)

print(f"输入形状: {x.shape}")
print(f"输出形状: {out.shape}")
print(f"每个 token 选择的专家（前2个样本的第1个token）:")
print(f"  样本1-token1 → 专家 {expert_choices[0, 0].tolist()}")
print(f"  样本2-token1 → 专家 {expert_choices[1, 0].tolist()}")
# 观察：不同 token 选择了不同的专家组合，这就是"稀疏激活"
```

#### 3.3 计算模型训练的 Compute Budget（FLOPs 估算）

```python
"""
功能：估算训练大模型所需的 FLOPs，验证 Chinchilla 配比
依赖：无（纯 Python）
"""

def estimate_training_flops(
    num_params: float,    # 模型参数量（单位：B，十亿）
    num_tokens: float,    # 训练 token 数（单位：T，万亿）
) -> dict:
    """
    估算训练 FLOPs（Hoffmann et al. 2022 公式）
    经验规律：训练 1 个 token × 1 个参数 ≈ 6 FLOPs（前向+反向）
    """
    params = num_params * 1e9
    tokens = num_tokens * 1e12
    total_flops = 6 * params * tokens

    # Chinchilla 最优配比：tokens ≈ params × 20
    optimal_tokens = num_params * 20  # 单位：B tokens

    return {
        "参数量": f"{num_params}B",
        "训练 tokens": f"{num_tokens}T",
        "总 FLOPs": f"{total_flops:.2e}",
        "Chinchilla 最优 tokens": f"{optimal_tokens:.0f}B",
        "是否训练充分": "✅ 足够" if num_tokens * 1000 >= optimal_tokens else "⚠️ 训练不足",
    }

models = [
    ("GPT-3",    175,  0.3),   # 175B 参数，0.3T tokens
    ("Chinchilla", 70, 1.4),   # 70B 参数，1.4T tokens
    ("LLaMA 3-8B",  8, 15.0),  # 8B 参数，15T tokens（过训练）
    ("LLaMA 3-70B", 70, 15.0), # 70B 参数，15T tokens（过训练）
]

print(f"{'模型':<15} {'参数':<10} {'tokens':<8} {'FLOPs':<15} {'最优tokens':<12} {'状态'}")
print("-" * 75)
for name, params, tokens in models:
    r = estimate_training_flops(params, tokens)
    print(f"{name:<15} {r['参数量']:<10} {r['训练 tokens']:<8} "
          f"{r['总 FLOPs']:<15} {r['Chinchilla 最优 tokens']:<12} {r['是否训练充分']}")

# 观察：GPT-3 训练不足，LLaMA 3 是故意"过训练"（推理更便宜）
```

#### 5.1 基础题

**题目 1**：根据 Chinchilla Scaling Law，一个 70 亿参数（7B）的模型"最优"应该训练多少 token？LLaMA 2-7B 实际训练了 2 万亿 token，这比最优多还是少？为什么这样做？

<details>
<summary>💡 参考答案（点击展开）</summary>

**评分标准**：
- ⭐ 算出 Chinchilla 最优 token 数：1 分
- ⭐⭐ 判断 LLaMA 2 是否过训练：2 分
- ⭐⭐⭐ 解释过训练的工程动机：3 分

**参考答案**：

Chinchilla 最优比 ≈ 参数量（B）× 20（B tokens）
7B × 20 = 140B tokens = **1400 亿 token**

LLaMA 2-7B 训练了 2 万亿（2T = 2000B）token，是 Chinchilla 最优的约 **14 倍**，属于严重"过训练"。

**为什么过训练**：
- 训练成本是**一次性的**，推理成本是**持续的**（每次调用都要花钱）
- 过训练小模型（7B）让它达到"正常大模型"（如 70B）的性能
- 推理时用 7B 而非 70B，每次调用速度快 10 倍、成本低 10 倍
- 结论：**把训练成本转化为推理效率**，是面向大规模部署时的合理选择

**常见错误**：以为 Chinchilla 给出的是"最高性能"配比——实际上是"最优 FLOPs 效率"配比，不是"性能最高"配比。
</details>

**题目 2**：MoE 模型的总参数量是 671B，但每次推理只激活 37B——这意味着什么？MoE 和 Dense 671B 模型相比，有什么优势和劣势？

<details>
<summary>💡 参考答案（点击展开）</summary>

**评分标准**：
- ⭐ 理解激活参数 vs 总参数：1 分
- ⭐⭐ 对比推理成本：2 分
- ⭐⭐⭐ 说出 MoE 的劣势：3 分

**参考答案**：

**含义**：671B 总参数被分布在许多"专家"网络中，门控机制每次只选择约 37B 参数的专家处理当前 token——推理时的实际计算量等同于一个 37B 的 Dense 模型。

**vs Dense 671B**：

| 维度 | MoE 671B | Dense 671B |
|------|---------|-----------|
| 推理 FLOPs | ≈ 37B | ≈ 671B（18× 更慢）|
| 推理延迟 | 低 | 高 |
| 显存（加载全部参数） | 671B × dtype | 671B × dtype（相同）|
| 模型容量（学习能力） | 高（671B 参数存储知识）| 高（相同）|
| 训练稳定性 | 更难（负载均衡问题）| 更容易 |

**MoE 的劣势**：
1. 必须把所有 671B 参数加载到 GPU 显存（或分布到多卡），对硬件要求高
2. 训练时负载不均衡问题（某些专家被过度使用）需要辅助损失处理
3. 小批量推理时，专家并行的优势不明显

**一句话总结**：MoE 是"用存储空间换计算速度"——显存要求和 Dense 671B 一样大，但推理速度像 37B。
</details>

#### 5.2 🔥 面试真题

**真题 1**（来源：[牛客网-2025-Q1-推理框架岗]）

> 什么是 KV Cache？在自回归推理中它解决了什么问题？有什么代价？

<details>
<summary>💡 参考答案（点击展开）</summary>

**评分标准**：
- ⭐ 解释 KV Cache 的功能：1 分
- ⭐⭐ 说明自回归推理的重复计算问题：2 分
- ⭐⭐⭐ 量化显存代价 + 优化方向：3 分

**参考答案**：

**问题背景**：自回归生成中，生成第 t 个 token 时，Attention 需要对**全部 t-1 个历史 token** 计算 Key 和 Value。若不缓存，每步都要重新计算，复杂度 O(n²)。

**KV Cache 方案**：把每一层 Transformer 中每个已生成 token 的 K/V 向量缓存在 GPU 显存中。生成新 token 时，只需计算新 token 的 Q/K/V，然后拼接历史 K/V 做 Attention——复杂度降至 O(n)。

**显存代价**（以 LLaMA 3-8B 为例）：
```
KV Cache 大小 = 2 × L × H_kv × d_head × T × dtype_bytes
= 2 × 32层 × 8头（GQA）× 128维 × 序列长度 × 2字节（FP16）
≈ 0.5MB/token
128K tokens ≈ 64GB 显存（仅 KV Cache）
```

**优化方向**：
- Paged Attention（vLLM）：非连续内存分配，提高多请求并发
- KV Cache 量化（INT8/INT4）：用精度换显存，减少 2-4 倍占用
- Sliding Window Attention：只缓存最近 N 个 token 的 KV

**常见错误**：认为 KV Cache 减少了总 FLOPs——实际上 FLOPs 没变，减少的是**重复计算量**（时间）和**显存读写**（带宽）。
</details>

**真题 2**（来源：[常见考点-B级]）

> 请解释 GQA（Grouped Query Attention）的原理，它相比 MHA 和 MQA 各有什么优缺点？

<details>
<summary>💡 参考答案（点击展开）</summary>

**评分标准**：
- ⭐ 解释三种 Attention 的 KV 头数差异：1 分
- ⭐⭐ 分析 KV Cache 显存影响：2 分
- ⭐⭐⭐ 说明为什么 GQA 是现代首选 + 举代表模型：3 分

**参考答案**：

三种方式在 Query 头数 h=32 时的 KV 头数：

| 方式 | KV 头数 | KV Cache 大小 | 质量 |
|------|---------|-------------|------|
| MHA | 32（与 Q 相同）| 最大（基准 1×）| 最好 |
| GQA | G（如 8）| 1/4 × MHA | 接近 MHA |
| MQA | 1 | 1/32 × MHA | 明显下降 |

GQA 把 h=32 个 Query 头分成 G=8 组，每组 4 个 Query 头共享 1 个 KV 头。相比 MHA KV Cache 减少 4 倍，相比 MQA 质量更好。

**为什么是现代首选**：MQA 质量损失太大（实测在多个基准上下降约 1-3%）；GQA 在质量（接近 MHA）和效率（接近 MQA）之间取得最佳平衡。LLaMA 3、Mistral、Qwen 2.5 均采用 GQA。

**常见错误**：把 KV 头数和 Query 头数混淆——GQA 中 Query 头数不变，只减少 KV 头数。
</details>

**真题 3**（来源：[牛客网-2024-Q4-大厂算法岗]）

> 如果给你 $10M（约 7000 万人民币）的算力预算，训练一个语言模型，你会如何分配参数量和数据量？请结合 Chinchilla Scaling Law 分析。

<details>
<summary>💡 参考答案（点击展开）</summary>

**评分标准**：
- ⭐ 提到 Chinchilla 配比：1 分
- ⭐⭐ 估算参数量和数据量：2 分
- ⭐⭐⭐ 考虑部署场景，讨论是否过训练：3 分

**参考答案**：

**基本估算**：
- $10M ≈ 约 10^23 FLOPs（基于 A100 成本估算）
- Kaplan Scaling：`C = 6ND`（前向+反向共 6 FLOPs/参数/token）
- Chinchilla 最优：`N* ∝ C^0.5`，`D* ∝ C^0.5`（等比增长）
- 代入：`N* ≈ sqrt(10^23 / 6) ≈ 4×10^10`，即约 **40B 参数**
- 最优数据量：`D* = N* × 20 ≈ 800B tokens`

**实际策略建议**（考虑部署）：
- 如果**部署规模大**（大量并发）：选更小参数 + 更多数据（如 20B 参数 + 2T tokens），换取更低推理成本
- 如果**部署规模小**：按 Chinchilla 最优（40B + 800B tokens）
- 如果**需要最高性能**：考虑 MoE 架构，同样算力可以做更大的有效模型

**加分点**：提到数据质量过滤（高质量 500B tokens 优于低质量 2T tokens）、以及 Test-Time Scaling 作为训练 Scaling 的替代方向。
</details>

#### 5.3 产品设计题

> 你的公司要为 B 端企业客户提供私有化部署的 LLM 服务（客户的数据不能上云）。客户预算有限：服务器只有 2 × A100（每张 80GB 显存，共 160GB）。你需要在以下模型中选一个并设计部署方案：① LLaMA 3-70B（Dense，FP16 需约 140GB）；② Qwen 2.5-72B（Dense，FP16 需约 145GB）；③ Mixtral 8×7B（MoE，全量加载需约 95GB）。请分析三个方案的利弊，给出推荐选择和关键配置参数。

**思考方向**：
- 70B Dense 模型 FP16 刚好超出显存 → 需要量化（INT8 约 70GB，INT4 约 35GB）
- Mixtral 8×7B MoE 总参数 46.7B，全量 FP16 约 95GB ✅ 可以装下
- 量化带来精度损失 vs MoE 的推理效率优势
- KV Cache 显存也要计入

#### 5.4 开放思考题

> Test-Time Scaling（推理时 Scaling）被认为是突破训练 Scaling 瓶颈的新方向——让模型在推理时"多想一会儿"（更长的 CoT）来替代更大的参数量。你认为这个方向的上限在哪里？它会完全替代训练时的 Scaling 吗？

**思考方向**：推理计算和训练计算的本质区别；"想更久"能替代"知道更多"吗；成本结构变化对产品的影响。
